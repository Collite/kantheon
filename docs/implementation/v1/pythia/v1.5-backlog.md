# Pythia v1.5+ Backlog

> Items explicitly deferred from v1 during the 2026-05-04 brainstorm. Each entry says **what** is deferred, **why** it was deferred, and **when to revisit** (the trigger that should bring it back).
>
> This document complements [`../../../design/pythia/Pythia-v1-Design.md`](../../../design/pythia/Pythia-v1-Design.md) §7 (Roadmap), which sequences the major phase deliveries. The backlog here is finer-grained — individual decisions deferred during the brainstorm.

---

## Deferred to Pythia v1.5

### Conceptual layer (`cnc`) integration in Themis

**What.** Themis consumes the `cnc.role` schema (concept "revenue" → role "metric" → which entities/attributes hold it). Today (v1) Themis is pinned to E-R model.

**Why deferred.** The `cnc` layer is platform work in flight (G2 in the Analytical Agent doc). Pythia v1 ships against E-R; the cnc-aware enrichment is additive. Themis can consume both layers without breaking changes.

**When to revisit.** When `cnc.role` schema and `er2cncRole` mappings are live in `metadata-mcp`. Trigger: platform Phase 2.2 (model expressiveness) ships.

### Self-hosted CHEAP embedding model

**What.** A locally-hosted embedding model (BGE-M3 or similar) for the EMBEDDING/CHEAP slot. Today (v1) Pythia uses Azure OpenAI for both EMBEDDING/STRONG and EMBEDDING/CHEAP — same model.

**Why deferred.** Self-hosting is operations effort; Azure OpenAI works for v1 even if it's not cheap. The cost matters at scale.

**When to revisit.** When per-investigation embedding cost becomes a budget pressure (likely once Pythia is processing 100+ investigations/day with heavy RAG).

### `model.decompose.variance` for honest RCA explained-variance

**What.** Metis capability that performs regression-style variance attribution: given a target metric and a set of factor handles, returns a decomposition with each factor's contribution. Today (v1) Pythia uses a heuristic (sum of supported-hypothesis explanatory_power, capped at 1.0).

**Why deferred.** Variance decomposition is non-trivial DS work; heuristic is "good enough" for v1's RCA use cases. Honest decomposition matters more for high-stakes RCA (board packs, regulatory analysis).

**When to revisit.** When Metis agent has bandwidth, and when the first request comes for a high-stakes RCA where heuristic confidence isn't acceptable. Pythia switches over by registering `model.decompose.variance` in capabilities-mcp; the planner uses it automatically.

### Per-engine IN-list materialise threshold

**What.** `query-mcp` exposes per-dialect (MS SQL / DuckDB / Polars) hints for the maximum efficient IN-list size. Pythia uses these to tune the materialise-vs-stay-hot decision. Today (v1) one default threshold (500) for all engines.

**Why deferred.** 500 is conservative across engines; Pythia works correctly without per-engine tuning. The optimisation is performance, not correctness.

**When to revisit.** When a Pythia investigation is observably slowed by either over-materialising small IN-lists or choking large ones in a specific engine.

### Learned per-task-kind token estimates

**What.** Replace configured constants for `task_kind → token estimate` with per-deployment moving averages computed from actuals. Today (v1) Pythia uses configured constants in YAML.

**Why deferred.** Constants are accurate enough for budget projection (within 2x is fine). Learning adds a small data store + update logic. Net value is marginal for v1 traffic levels.

**When to revisit.** When budget projection error becomes a real cost (frequent over- or under-estimation hurting investigations).

### Interactive plan editing UX at approval

**What.** When a plan is presented for approval, the user can promote/demote/add/reorder hypotheses, not just APPROVE / REJECT_WITH_COMMENT. Today (v1) approval is APPROVE | REJECT_WITH_COMMENT (re-plan with comment as additional context).

**Why deferred.** REJECT_WITH_COMMENT covers most of the value with a much simpler UX. Full editing is Iris-frontend work, not Pythia-internal — better landed once user feedback identifies which edits matter most.

**When to revisit.** When users repeatedly want a specific edit pattern (the data signal is in their REJECT_WITH_COMMENT messages).

### Suggested-followup as one-click child-investigation seed (Iris UX)

**What.** Loose ends carry `suggested_followup` strings; Iris renders them as one-click "investigate this" buttons that spawn child investigations with the loose end as the seed. Today (v1) the field exists in the artifact and Iris renders it as plain text.

**Why deferred.** This is Iris frontend work, not Pythia-internal. The data structure supports it from v1; the UX can land any time.

**When to revisit.** Iris v2.x roadmap pickup.

## Deferred to Pythia v2

### Sweeping-simulation infrastructure

**What.** Investigations that fan out across a parameter grid: a parent investigation spawns N child investigations, each running the same simulation with different scenario_params; the parent aggregates results into a sensitivity-analysis artifact. Today (v1) one investigation = one set of scenario_params.

**Why deferred.** Fundamentally different shape: long-running (hours-to-days), multi-pod parallelism, heavyweight result aggregation. Needs new infrastructure (Temporal-class durable workflow engine, or equivalent).

**When to revisit.** When users explicitly request "run this scenario across price = [10, 20, 30, 40, 50] and tell me how margin sensitivity to price varies."

### Promote DAG executor from Kotlin coroutines to Temporal

**What.** Replace the custom coroutine-based DAG executor with a Temporal workflow. Today (v1) Pythia uses custom Kotlin coroutines + Postgres-backed checkpointer.

**Why deferred.** Temporal is heavyweight (separate cluster, separate operational concern, opinionated DSL). Custom coroutines fit Pythia v1's needs (single-Pythia, investigations measured in seconds-to-minutes). Temporal becomes essential if investigations regularly run for hours+, or if multi-Pythia coordination is required.

**When to revisit.** When sweeping simulations are on the v2 agenda, OR when single-investigation runtimes routinely exceed 30 minutes, OR when we need to coordinate work across multiple Pythia instances.

### Multi-Pythia coordination for partition-by-tenant scaling

**What.** Multiple Pythia instances coordinate work — investigations sharded by tenant, results aggregated. Today (v1) one Pythia handles all investigations.

**Why deferred.** Single Pythia is enough for v1 traffic. Sharding is real engineering effort.

**When to revisit.** When single Pythia is a throughput bottleneck.

### Per-call-site mid-flight tier escalation

**What.** During an investigation, the planner can escalate a specific call site (e.g., evaluator) from CHEAP to STRONG on its own initiative when it detects a hard case — within the investigation's budget cap. Today (v1) escalation is per-investigation only (set in the request).

**Why deferred.** Power feature; calibration risk (LLMs are bad at predicting when they need a stronger model). Per-investigation escalation covers the explicit use cases.

**When to revisit.** When CHEAP-tier evaluator quality is observably hurting RCA accuracy, AND we have telemetry to validate that mid-flight escalation would help.

## Open / not-yet-decided

### LLM orchestration framework choice (tension 5, second half)

**What.** Pythia's LLM-orchestration parts (planner, evaluator, synthesizer call patterns, prompt templates, structured-output handling) need a framework. Candidates: **Koog**, **Embabel**, **LangChain4j**, **Spring AI**, or roll-our-own thin wrapper.

**Why open.** The DAG executor decision (custom Kotlin coroutines + Postgres) is locked. The LLM orchestration is a separate decision and was explicitly deferred to a focused evaluation pass. The frameworks differ in: prompt-template ergonomics, structured-output / tool-use story, streaming support, multi-vendor abstraction, observability, and how they compose with Spring (since `llm-gateway` is Spring Boot).

**When to revisit.** Before Pythia v0 implementation starts in earnest. Belongs in `open-questions.md`.

### NARRATIVE_FRAGMENT vs. fold-into-synthesizer

**What.** v1 keeps `NARRATIVE_FRAGMENT` as a RenderNode kind: planner-declared per-fragment LLM calls (one CHEAP-tier call per fragment) for hypothesis-specific captions. Alternative: only the synthesizer ever produces text; everything is composed at synthesis time.

**Why kept in v1.** Bora confirmed (2026-05-04) — useful for hypothesis-specific captions tied directly to evidence steps, locality of reasoning.

**When to reconsider.** If NARRATIVE_FRAGMENT calls accumulate to a significant fraction of investigation cost, or if synthesis-time composition produces equivalent quality, this becomes a fold-target.
