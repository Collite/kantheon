## Grounding deltas (stage start, 2026-06-19)

- **No reusable streamable-HTTP MCP client exists** (verified — hebe's is stdio-only, no auth forwarding; no shared lib). Golem's `QueryClient` is **net-new**: MCP-SDK `Client` over a StreamableHTTP transport, forwarding `Authorization: Bearer`. Built Golem-local for now (no other consumer yet; consolidate to a shared `mcp-client` lib when Pythia needs it).
- **OBO bearer source is `AdmittedCaller.bearer`** (Stage 2.2 admission), **not** the `Caller` proto — `golem/v1 Caller` carries no token field. The `/v1/answer/sync` route admits the caller (already does bearer validation) and threads `AdmittedCaller.bearer` to the `QueryClient`. Never a service identity (kantheon-security §2).
- **theseus-mcp `query` tool**: input `source` + `source_language` (`sql|transdsl|…`) + `parameters` + `format=json` + `row_limit`; rows arrive in the `TextContent` (JSON array), metadata (`rowCount`, `columns`, `truncated`) in `structuredContent`. `compile` tool: `source`+`source_language`+`target_dialect` → `compiledSql` (the `compile_first` pre-check for FREE_SQL).
- **Envelope deterministic path is private** in envelope-render (`FormatCatalog.deterministicFallback`/`tableResult`). Phase-2 sync answers assemble a **minimal table `FormatEnvelope`** directly (content_json = rows + `FormatSpec{TABLE}` + `inferTableHeaders`). Full LLM format pipeline + chips/drilldowns are Phase 3.
- **ReasoningNode** execution (LLM reasoning) is Phase 3 — the Phase-2 executor runs Query nodes and renders tables; a ReasoningNode is recorded `SKIPPED`.

## Parity refresh (2026-06-24) — pattern-parametrization rail + row-detail selection

> **Why.** T1–T5 (above) predate the ai-platform Golem **pattern-parametrization** and **row-detail selection** arcs (plan.md §10 Δ1/Δ4). T1–T3 already pass a `parameters` field to `theseus-mcp query` (grounding delta above), so the *transport* is in place — but the **typed rail** (surface→wire type coercion, the verbatim-`sql_template`-with-`{name}` contract, the unfilled-param guard) and the **`resolve_selection` node + selection state** are net-new. Land these as part of closing Stage 2.4 (before the `golem/v0.1.0` tag).
>
> **Ordering: T7 (Proteus restoration) is the first slice — everything downstream depends on it.** The Theseus/Proteus typed-parameter path is **PARTIAL** (verified 2026-06-24; a fork regression). The MCP edge accepts `{value,type}` and the Brontes worker JDBC-binds by type, **but the forked Proteus orchestrator ignores `context.parameters` during translation**, so `{name}`-style named-parameter SQL (the Golem-pattern case) is **not** rewritten to `?` and reaches the worker un-bridged. Do T7 before the rail (T8); until it lands, the rail can send typed params but the named-param substitution won't happen. Tracked in plan.md §2 pre-flight + master-plan §7.
