# Stage 1.1 — Catalog conflict log

> **Produced by P1 Stage 1.1 T1.** The merge rule (architecture §4.2): **kantheon's pin wins** unless Hebe has a hard requirement, in which case kantheon bumps (recorded as an *exception*, flagged for Bora). No exceptions were needed — every Hebe pin is API-compatible at kantheon's (equal-or-lower) version.

## Method

Rather than rewrite the dependency block of all 21 Hebe module `build.gradle.kts` files, the merge **adds Hebe's catalog alias names** (`koog-utils`, `mcp-kotlin-sdk`, `otel-sdk`, `kotest-assertions`, the `ktor-server`/`ktor-client`/`testing` bundles, …) into kantheon's `gradle/libs.versions.toml`, pointing them at the **merged (kantheon-winning) version refs**. Where Hebe and kantheon already share an alias (e.g. `kotlinx-coroutines-core`, `flyway-core`, `jgit`), the Hebe modules resolve through kantheon's existing alias automatically — so the kantheon pin wins with **zero** per-module dependency churn. The only per-module edit in T3 is the `plugins {}` block (the retired `hebe.library`/`hebe.application` convention plugins).

## `plugin-template` — built module or packaged resource?

`agents/hebe/plugin-template/` is **not** a Gradle subproject of the standalone Hebe build (it is absent from `agents/hebe/settings.gradle.kts`'s `include(...)`). It is a **standalone scaffold** third parties copy out and build against the published `plugin-api`. It is therefore **not** added to the kantheon root `settings.gradle.kts`; it stays a packaged template. The "21 modules" count = the 21 `include(...)` entries in the old Hebe settings file (the 16 leaf areas with `channels` and `providers` and `tools` each expanded into their submodules). The plugin-template build is verified in Stage 1.2 T6 (the PF4J classname smoke test), not wired into the root build.
