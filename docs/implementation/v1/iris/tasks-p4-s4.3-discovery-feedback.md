## Grounding

- **Discovery:** `GET /v1/discover` → `{domains:[DomainCard]}` from the BFF capabilities cache, **filtered by the caller's `visibility_roles`** (`non_routable` excluded). `DomainCard = {agent_id, display_name, blurb (from description_for_router), example_questions:[str]}` (`example_questions` is the v1 source; `preferred_queries` lack display text → v1.1). The capabilities cache + role filter machinery exists from Phase 3 (`capabilities/` package, used by static chips + the re-ask picker).
- **Feedback:** `iris_feedback` table already exists (Phase 1 D6; DDL contracts §3.2) with `corrected_agent_id` written by Phase 3 Stage 3.2's `reask_agent`. This stage adds the `POST /v1/turns/{id}/feedback` upsert + FE 👍/👎 + the offline export.
- **Audit:** the hash-chained `iris_audit` writer (`audit/Audit.kt`, `infra/ExposedAuditStore.kt`, `Ed25519Signer`) + chain-verify unit test exist from Phase 1 (Stage 1.3 T7 / Stage 1.4 Group A). This stage adds the **verify endpoint**, **retention/archive job**, and the **signing-key runbook** — completing PD-8.
- **Export:** `just feedback-export` is a new recipe; per-agent schema adapters write to each agent's `eval/candidates/` directory. Themis's adapter turns wrong-agent + correction pairs (`corrected_agent_id`) into labeled routing examples.

## Tasks

- [ ] **T1 — `GET /v1/discover` (contracts §2.6) + FE discovery panel.** BFF: assemble `DomainCard`s from the capabilities cache, **filter by the caller's `visibility_roles`** (`non_routable` excluded — PD-8: you only discover what you may ask). FE: first-run / empty-session panel ("Here's what I can answer about ERP, HR, Investment…") + suggested-question chips on an empty input box; a question chip submits as a normal turn. **Tests first:** `DiscoverRoutesSpec` — role filter applied (user without HR role sees no HR card; `non_routable` excluded); `DomainCard` shape (blurb from `description_for_router`, `example_questions`). FE panel + chip-submit component spec.
- [ ] **T2 — Tests first + `POST /v1/turns/{turn_id}/feedback` (contracts §2.9).** Upsert per `(turn_id, user_id)` (users may change their mind): `{verdict:"up"|"down", reason?:"wrong_data"|"wrong_agent"|"wrong_format"|"too_slow"|"other", comment?}`. Capture is synchronous; **no agent sees feedback at runtime**. FE: 👍/👎 on every answer bubble + a one-tap reason picker on 👎. **Tests first:** `FeedbackRoutesSpec` — insert then update (same row, verdict changes); joins `iris_turns` for agent_id; non-owner → 404. FE feedback-control component spec.
- [ ] **T3 — `just feedback-export` + per-agent adapters.** A recipe + a small exporter reading `iris_feedback` (⋈ `iris_turns` question/agent_id, ⋈ `iris_audit` payload for `RoutingDecision`) and writing per-agent `eval/candidates/` files via per-agent schema adapters. Themis adapter: labeled routing examples from wrong-agent rows + `corrected_agent_id` correction pairs (the strongest misroute label). Human curation before promotion into gate corpora (documented, not automated). **Tests first:** exporter spec — a `wrong_agent` + `corrected_agent_id` row produces the expected Themis candidate record; per-agent routing of records by `agent_id`.
- [ ] **T4 — Audit completion: verify endpoint + retention + runbook.** `GET /v1/audit/verify?segment=YYYY-MM` (gated on the `iris.audit.verify-role` realm role, default `iris-admin`) recomputing the chain over a monthly segment (each segment anchored independently to the prior segment's terminal hash) and reporting first-break or OK. A retention/archive job: when `iris.audit.retention-months` elapses, archive + drop **whole monthly segments**, keeping later segments verifiable. Write the signing-key **runbook** (rotate = new Secret + restart; verify each rotated-out segment before archiving). **Tests first:** verify-endpoint spec (intact chain → OK; a tampered payload → first-break seq reported; non-admin → 403); whole-segment retention spec (older segment archived/dropped, newer segments still verify).
- [ ] **T5 — Metrics + Grafana extension.** Emit `feedback_total{agent_id, verdict, reason}` + discover counters here. The inbox metrics (`iris_inbox_open_total`/`iris_inbox_reattach_total`/`iris_lifecycle_nats_connected_gauge`) were emitted in Stage 4.1, the artifact metric (`iris_artifact_refresh_total{kind, result}`) in Stage 4.2, the investigate-chip counters (`iris_investigate_chip_shown_total`/`_clicked_total`) in Stage 3.3, and the audit counters (`iris_audit_write_total`/`iris_audit_chain_verify_failures_total`) in Stage 3.3 — this task only adds the two new families and the Grafana rows; do not re-emit the others. Extend the Stage-3.3 Grafana `iris-bff` dashboard with feedback, inbox, and artifact-refresh rows. **Tests first:** metrics spec for `feedback_total` increment on a feedback POST.
