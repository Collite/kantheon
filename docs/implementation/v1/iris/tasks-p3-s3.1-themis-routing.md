# Iris Phase 3 Stage 3.1 — Themis integration + routing dispatch

> **Goal (plan §5 Stage 3.1).** The BFF resolves every turn through Themis before dispatch: `ThemisClient.understand()` per turn (profile `CHAT_QUICK`, excerpt + entity context in, **user bearer forwarded**), `HandoffContext` assembled on every dispatch (PD-1), an `AgentDispatcher` keyed by `chosen_agent_id`, the `RoutingPickChip` round-trip (`needs_user_pick` → chips → re-issue with `routing_hint`), `RefusalWithGaps` rendering, and the PD-13 decomposition rule. **Closing this stage closes Themis Stage 3.6 tasks 2–3** (the "Iris BFF stub" — single implementation here).

## Grounding (codebase as of 2026-06-23, post-2.4 Group A)

The dispatch flow today is **Themis-less** — `ChatDispatcher.runTurn` goes straight to `GolemV2Client`:

- `api/ChatDispatcher.kt` — `runTurn(caller, sessionId, question, desiredFormat, correlationId, emit)`: `ensureThread` → `mux.run(turnId, client.chatStream(V2ChatRequest(...), caller.userId, corr, caller.bearer), emit)` → `persist(...)` (appends `iris_turns` row + writes the `turn` audit event). **This is the seam Themis slots into**: resolve *before* `client.chatStream`, pass the chosen agent + handoff down.
- `api/ChatRoutes.kt` drives `/v1/chat/stream`, `/v1/chat/turn`, `/v1/chat/resume`; `api/ChatDtos.kt`'s `ChatTurnRequestDto` today has **only** `{sessionId, question, desiredFormat?}` (confirmed 2026-06-23). This stage **adds** `routingHintAgentId: String? = null` (chip-click reissue). The `origin`/`originRef`/`dryRun` fields from contracts §1.2 are out of scope here — add `routingHintAgentId` only.
- `domain/ConversationExcerpt.kt` builds the last-N-turns excerpt (Stage 1.3). `domain/Records.kt` holds `NewTurn`, `TurnStatus`, `TurnPointer`-equivalents; `SessionStore.appendTurn` persists. `TurnPointer.current_view` / `applied_context` columns exist in the proto + `iris_turns` is the snapshot table — but nothing populates `current_view`/`applied_context` yet (PD-1/PD-4 wiring starts here, completes in 3.2).
- `api/Auth.kt` → `CallerIdentity { userId, tenantId, bearer, roles? }` (the forwarded bearer is already on the caller — `client.chatStream(..., caller.bearer)`).
- No `routing/` package yet (architecture §3.1 reserves it). No `ThemisClient`. No `AgentDispatcher` (dispatch is hard-wired to `GolemV2Client`).
- `shared/libs/kotlin/capabilities-client` → `CapabilitiesReadClient` (display names, manifests) is available for chip labels.

**Design decisions for this stage (contract-faithful):**

1. **Resolve-then-dispatch.** Introduce `routing/ThemisClient` + `dispatch/AgentDispatcher`; `ChatDispatcher.runTurn` calls `themis.understand(...)` first, branches on the `ResolveResponse.outcome` oneof, and only on `Resolution` (or a `routing_hint` Layer-0 path) calls the dispatcher. `AwaitingClarification`/`RefusalWithGaps` short-circuit to chips/error envelopes — no agent call.
2. **AgentDispatcher is a registry of clients keyed by `agent_id`.** At Phase 3 the only registered client is `golem-v2` (transitional); the map is the seam native Golem/Pythia clients plug into later. An unknown `chosen_agent_id` → error envelope `code=NO_AGENT_CLIENT`.
3. **`routing_hint` Layer-0 path.** When `ChatTurnRequest.routingHintAgentId` is set (RoutingPickChip re-issue), it rides into `ResolveRequest.routing_hint`; Themis honours it as Layer 0 and returns a `Resolution` with that `chosen_agent_id`. The BFF does not bypass Themis — it lets Themis validate the hint (per themis contracts: `picked_agent` re-entry / `routing_hint`).
4. **Profile = `CHAT_QUICK`** always at the Iris edge (Pythia's `INVESTIGATION_DEEP` is reached via escalation/InvestigateChip, not the default turn).

## Tasks

- [ ] **T1 — Tests first: `ThemisDispatchSpec` (unit, Kotest + MockK).** Define the dispatch decision table against a `FakeThemisClient` returning canned `ResolveResponse`s. Cases:
  - `Resolution{chosen_agent_id=golem-erp, routing.needs_user_pick=false}` → dispatch to that agent; one envelope out; `iris_turns.agent_id = golem-erp`.
  - `Resolution{routing.needs_user_pick=true, alternates=[pythia(why=…), golem-erp(why=…)]}` → **no** agent call; a single `FormatEnvelope` whose `chips` are `RoutingPickChip`s (one per alternate, `agent_id`/`label`/`why`); turn persisted with `alternatesOffered`.
  - `AwaitingClarification{multi_question, decomposition=SPLIT, sub_questions=[…]}` → decomposition chips (one `PromptChip` per sub-question), no dispatch.
  - `AwaitingClarification{multi_question, decomposition=KEEP_TOGETHER}` → proceed whole (single dispatch); `decomposition_rationale` surfaced as a hint (`ResponseMessage` / text bubble).
  - `RefusalWithGaps{gaps:[NO_ENTITLED_AGENT]}` → error envelope `error_code=NO_ENTITLED_AGENT`, gap descriptions rendered, **no** dispatch.
  - `RefusalWithGaps{gaps:[CAPABILITY_UNAVAILABLE/OUT_OF_DATA_SCOPE/…]}` → error envelope with the gap list + `suggested_action`s.
